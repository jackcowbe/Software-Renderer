# 畫線演算法 (Line Drawing Algorithm)

簡單說明這一張要說明的內容，在實作軟體渲染之前，我們得學會怎麼在螢幕上畫一條線，下面會逐一介紹我製作畫線演算法的過程。

##第一次嘗試 時間步進演算法

在螢幕上畫出一條線 最簡單的方法就是把X軸當成是我們的時間點，根據時間的流動畫出一條線。

在這張圖片裡，我們可以看到這是一條 Y=2X 的直線。我們來思考，隨著時間推進，X 會越來越大。我們必須確認在當前時間下，X 和 Y 分別在哪裡。所以用數學公式來表達這個「走格子累積」的過程：

#### ⏰ 時間的累加
當前總時間，本質上就是把每一步「微小的步伐時間」通通加總起來：

$$\text{當前總時間} = \sum_{i=1}^{\text{總步數}} \text{微小步伐時間}_i$$

#### 📍 位置的累積
而最終的像素坐標位置，就是從起點位置出發，加上每一步主軸前進時，副軸被迫跟著等比例抬高的「微小高度變動」的總和：

$$\text{最終像素位置} = \text{起點位置} + \sum_{i=1}^{\text{總步數}} (\text{微小高度變動} \times \text{主軸微小位移}_i)$$

說穿了，數學上的 $\sum$（加總符號），在我們 C++ 程式碼的肉體裡，其實就只是迴圈裡最簡單的 `+=` 連續累加而已！

####我的畫線演算法

我知道現在在許多軟體渲染的教學上已經有許多現成的演算法(Bresenham, DDA)等，但我認為如果只是去理解現成的演算法，過程其實過於簡單，而且只是被動的接收現有的技術，於是我決定自己解決這個畫線的問題，並使他的效能盡可能的完美。

####原畫線演算法的效能問題

原本的時間步進演算法在繪製線段上絕對是沒有問題，但是我們的時間如果少於我們的斜率，會因為採樣點(時間上限)過於少造成線段斷裂，且原本的方法在處理連續的線段絕對是沒有問題，但原因是我們的螢幕是像素是離散化的。所以我們容易因為在同一個像素上過度繪製重複的顏色造成算利浪費。
例如我們的像素分為1,2,3,4。但在這個演算法中時間是無限細分的，假如X=0.01是在像素0，但我們沒有像素0這個像素，所以我們只能不繪製。他需要持續運算到X=0.5才會繪製在像素1兒0.5到1之間又需要運算無數次。
但我認為這個方法並沒有不好，原因只是我們不需要這麼高精度的線段。我們在像素上繪製出來的線段是有鋸齒得。原本的演算法只是過於精準了。於是我開始想解決這個問題的方法。

####嘗試解決

這中間我腦袋冒出了無數個天真想法，我想像能不能有一個垂直於X向量的力來決定他的斜率，而像素會像因為被拉扯而在關節上(Y軸)位移。實作下來這是非常不OK的。在接觸軟體渲染前我接觸了一段間的HLSL，我下意識地想使用step函式。但我在想一個好的演算法是在任何硬體上都適用的，
於是如果硬體做浮點運算會是一個昂貴的開銷，那這個想法會變得不切實際。於是我又想像出無數個天真地的想法。最後我得出了答案，把線段看成頻率並找出他的週期，靠線段的長度來裁減這段頻率。

####實作演算法

我想我能把斜率想像成是我們最終的偏移量，所以我們必須讓所有X軸的像素共同承擔他這個偏移量。簡化一下，我們先思考今天有10個蘋果(Y軸分量)分給三個小朋友(x軸)，一個小朋友能分到幾個蘋果?答案是3個，但會多出一個蘋果。但這樣變成一個小朋友必須負擔三個蘋果。這樣不行。
所以我們得讓比較大的分量去承擔較小的偏移。換個想法，今天只有3個蘋果分給10個小朋友。那便是1個小朋友會被分到1/3的蘋果。3個小朋友1個蘋果。第10個一邊去，喔不是，第10個小朋友需要跟另外2個不能參與的共用一個不存在的第4顆蘋果。
但我們這個方法主要是找出周期和規律，我們想像有無數個小朋友和蘋果。但我們只取第幾個到第幾個(線段長度)小朋友(像素)，看看他們有幾個一起承擔一個蘋果。所以我們是數小朋友(像素)看他們幾個要偏移一個蘋果(偏移分量)。

我們先用數學表示:

📥 1. 初始化階段（每步拿多少？）

要把副軸總共要跨越的蘋果數，平均分攤給主軸總共前進的小朋友數。初始化的離散帶餘除法公式如下：

隨後，利用整數除法與取餘數（模運算）將其解構：

$$\text{每步固定走幾格} = \frac{\text{副軸總共要跨越的蘋果數}}{\text{主軸總共前進的小朋友數}} \quad \text{（無條件捨去取整數）}$$

$$\text{每步多出來的殘留誤差} = \text{副軸總共要跨越的蘋果數} \pmod{\text{主軸總共前進的小朋友數}}$$

🔄 2. 迴圈核心步進（下一個像素點在哪？）

進入光柵化核心迴圈後，主軸每前進一步，誤差聚寶盆就會動態累加殘留誤差：

$$\text{當前誤差累積} = \text{上一步的誤差累積} + \text{每步多出來的殘留誤差}$$

當聚寶盆裡的殘留誤差大於或等於主軸總共前進的小朋友數時，觸發動態補償，副軸強行跨越一步，並扣除一個分配週期：：

$$\text{當當前誤差累積} \ge \text{主軸總共前進的小朋友數 時：}$$

$$\text{副軸位置} = \text{副軸位置} + 1$$

$$\text{當前誤差累積} = \text{當前誤差累積} - \text{主軸總共前進的小朋友數}$$

接下來我們用程式表示:

【初始化階段：先算好每一步怎麼分】
每步固定走幾格 = 主軸總共前進的小朋友數 / 副軸總共要跨越的蘋果數 （取整數商數）
每步多出來的殘留誤差 = 主軸總共前進的小朋友數 % 副軸總共要跨越的蘋果數 （取除法餘數）

當前誤差累積 = 0
當前副軸位置 = 起始點位置

【主循環階段：主軸（小朋友）開始一步一步往前走】
從 主軸起始點 一路走到 主軸終點：
    
    在螢幕上點亮像素格點 (當前主軸位置, 當前副軸位置)
    
    // 基礎分配：副軸先走每一步固定要分擔的格子
    當前副軸位置 = 當前副軸位置 + 每步固定走幾格
    
    // 誤差累積：把分不平的殘留誤差（多出來的蘋果）丟進聚寶盆
    當前誤差累積 = 當前誤差累積 + 每步多出來的殘留誤差
    
    // 溢位動態修正：如果聚寶盆滿了，分不平了，副軸強行多跨 1 格！
    如果 當前誤差累積 >= 主軸總共前進的小朋友數 則：
        當前副軸位置 = 當前副軸位置 + 1
        當前誤差累積 = 當前誤差累積 - 主軸總共前進的小朋友數 // 減去一個週期，重新累積
        
    主軸位置 = 主軸位置 + 1 // 下一個小朋友進場

接下來是英文的虛擬碼

    void line2(int2 Start_Vertex, int2 End_Vertex) {
    // ------------------------------------------------------------------------
    // 1. 幾何量測階段：計算兩點在水平（X）與垂直（Y）方向的原生向量位移量
    // ------------------------------------------------------------------------
    int X_Vector = End_Vertex.x - Start_Vertex.x; // 計算水平方向的純向量位移
    int Y_Vector = End_Vertex.y - Start_Vertex.y; // 計算垂直方向的純向量位移

    // ------------------------------------------------------------------------
    // 2. 絕對值化階段：利用 abs() 取得兩軸的「純物理跨越長度」（即總蘋果數與總人數的原型）
    // ------------------------------------------------------------------------
    int lengthX = abs(X_Vector); // 取得水平方向的絕對跨越長度
    int lengthY = abs(Y_Vector); // 取得垂直方向的絕對跨越長度

    // ------------------------------------------------------------------------
    // 3. 主副軸判定階段：利用 max() 找出誰是主導迴圈前進的「主軸」
    // ------------------------------------------------------------------------
    // 如果 lengthX 較長，則 X 是主軸，Y 是副軸（把 Y 個蘋果分給 X 個小朋友）
    // 如果 lengthY 較長，則 Y 是主軸，X 是副軸（把 X 個蘋果分給 Y 個小朋友）
    int length = max(lengthX, lengthY); // 找出最長的一軸，作為主循環的總步數（總人數）

    // 防禦機制：若起點與終點完全重合（長度為 0），直接點亮起點並原地結束
    if (length == 0) {
        SetPixel(int2{Start_Vertex.x, Start_Vertex.y}, color{255, 255, 255}); // 點亮唯一的孤點
        return; // 收工返回
    }

    // ------------------------------------------------------------------------
    // 4. 方向步進判定階段：決定兩軸座標前進時是該 +1 還是 -1（處理任意八分方位的線段）
    // ------------------------------------------------------------------------
    int stepX = (X_Vector > 0) ? 1 : -1; // 若終點在右方則水平往 +1 走，否則往 -1 走
    int stepY = (Y_Vector > 0) ? 1 : -1; // 若終點在上方則垂直往 +1 走，否則往 -1 走

    // ------------------------------------------------------------------------
    // 5. 離散分攤初始化階段：根據誰是主軸，精確提取主軸與副軸的物理長度
    // ------------------------------------------------------------------------
    int Major_Length = 0; // 用於存放主軸的物理總長度
    int Minor_Length = 0; // 用於存放副軸的物理總長度
    int Start_Vertex_Major = 0; // 用於存放主軸的起始點座標
    int Start_Vertex_Minor = 0; // 用於存放副軸的起始點座標

    if (length == lengthX) { // 狀況一：如果是 X 軸比較長（X 是主軸）
        Major_Length = lengthX; // 主軸長度設定為 X 的長度
        Minor_Length = lengthY; // 副軸長度設定為 Y 的長度
        Start_Vertex_Major = Start_Vertex.x; // 主軸起始座標綁定 X 起點
        Start_Vertex_Minor = Start_Vertex.y; // 副軸起始座標綁定 Y 起點
    } else { // 狀況二：如果是 Y 軸比較長（Y 是主軸）
        Major_Length = lengthY; // 主軸長度設定為 Y 的長度
        Minor_Length = lengthX; // 副軸長度設定為 X 的長度
        Start_Vertex_Major = Start_Vertex.y; // 主軸起始座標綁定 Y 起點
        Start_Vertex_Minor = Start_Vertex.x; // 副軸起始座標綁定 X 起點
    }

    // ------------------------------------------------------------------------
    // 6. 相位與步進頻率計算：對齊你原本的「定點數頻率累加器」邏輯
    // ------------------------------------------------------------------------
    // 將副軸長度左移 16 位元（放大成 Q16.16 定點數），除以總步數，算出每一步副軸應該分攤的「定點數步進頻率」
    int frequency = (Minor_Length << 16) / length; // 計算主軸每走一步，副軸動態累積的離散誤差頻率

    // 初始化副軸的相位累加器（聚寶盆），同樣左移 16 位元進入定點數宇宙
    // 加上 32768（即 0.5 的 Q16.16 表達法：1 << 15）是為了在底層實現「四捨五入」的離散網格修正
    int minor_phase_accumulator = (Start_Vertex_Minor << 16) + 32768; 

    // ------------------------------------------------------------------------
    // 7.利用標準 for 迴圈伴隨時間時間推進
    // ------------------------------------------------------------------------
    for (int t = 0; t <= length; t++) { // 從時間 0 一路推進到主軸總長度
        
        // 提取當前副軸位置：將定點數相位累加器算術右移 16 位元，解碼回真實的螢幕像素整數格點
        int current_minor = minor_phase_accumulator >> 16; 
        
        // 提取當前主軸位置：起點座標隨著時間 t 以及正確的步進方向（+1/-1）線性推進
        // 注意：這裡為了對應你正向/反向線段的 step 權重，將時間 t 乘以步進方向
        int current_major = Start_Vertex_Major + (t * ((length == lengthX) ? stepX : stepY)); 

        // 座標重組還原：根據目前誰是主軸，把主軸與副軸的整數座標正確塞回實體 X 與 Y
        int x = 0; // 宣告最終要點亮的實體 X 座標
        int y = 0; // 宣告最終要點亮的實體 Y 座標

        if (length == lengthX) { // 如果當前是 X 當主軸
            x = current_major; // 實體 X 承接主軸軌道
            y = current_minor; // 實體 Y 承接副軸軌道
        } else { // 如果當前是 Y 當主軸
            x = current_minor; // 實體 X 承接副軸軌道
            y = current_major; // 實體 Y 承接主軸軌道
        }

        // 點亮當前像素：在畫布上精確噴出這一發經過四捨五入修正後的晶瑩剔透網格
        SetPixel(int2{x, y}, color{255, 255, 255}); 

        // 誤差累積：相位累加器吃進每一步分不平的殘留定點數頻率（多出來的蘋果量）
        // 如果是反向步進，頻率累加方向需配合副軸的 step 方向進行符號權重修正
        minor_phase_accumulator += frequency * ((length == lengthX) ? stepY : stepX); 
    }
    }
